# How to Register Custom Item IO Methods

[简体中文](../zh_cn/custom-item-io-method.md) **English**

> Generated by GLM-5.1

## Overview

HexParse manages Iota read/write operations for items through the `IOMethod` class. When a player holds an item and performs parsing/writing, HexParse looks up the `IOMethod` matching that item type to execute data I/O.

Through `HexParseAPI.CreateItemIOMethod`, you can register IO methods for custom items, enabling HexParse to read Iotas from or write Iotas to your items.

## How IOMethod Works

`IOMethod` automatically registers itself into the global map `ITEM_IO_TYPES` upon construction, using the item's `Class` as the key. When I/O is needed, `IOMethod.get()` looks up the corresponding `IOMethod` based on the item instance's `Class`.

### Lookup Priority

When a player holds items, HexParse checks both main hand and offhand:

1. Look up `IOMethod` for both main hand and offhand items
2. If both have an `IOMethod`, choose the one with lower `priority` (smaller number)
3. If `priority` is equal, prefer the offhand

### Validator

The `validator` parameter allows you to dynamically determine at runtime whether a specific item instance can be used for read/write operations:

```kotlin
fun validator(stack: ItemStack, isWrite: Boolean): Boolean
```

- `stack`: The current item stack instance
- `isWrite`: `true` for write operations, `false` for read operations
- Returns `false` to indicate the item instance does not support this operation

## API Method

### CreateItemIOMethod

```kotlin
HexParseAPI.CreateItemIOMethod(
    cls: Class<*>,
    writer: ((ItemStack, CompoundTag) -> Unit)? = null,
    reader: ((ItemStack) -> CompoundTag?)? = null,
    priority: Int = 0,
    validator: ((ItemStack, Boolean) -> Boolean)? = null,
)
```

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `cls` | `Class<*>` | — | The `Class` object of the item, used for matching |
| `writer` | `((ItemStack, CompoundTag) -> Unit)?` | `null` | Write function that writes NBT data into the item |
| `reader` | `((ItemStack) -> CompoundTag?)?` | `null` | Read function that reads NBT data from the item |
| `priority` | `Int` | `0` | Priority; lower numbers mean higher priority |
| `validator` | `((ItemStack, Boolean) -> Boolean)?` | `null` | Validation function that determines whether the item instance is usable |

### Default Behavior for writer and reader

- **writer is `null`**: The default behavior writes NBT data to the item's `data` tag (suitable for items implementing `IotaHolderItem`)
- **reader is `null`**: The default behavior calls `(item as IotaHolderItem).readIotaTag(stack)` (suitable for items implementing `IotaHolderItem`)

## Built-in IOMethod Registrations

HexParse has registered `IOMethod` for the following HexCasting items:

| Item Class | writer | reader | Description |
|------------|--------|--------|-------------|
| `ItemFocus` | Default (`IotaHolderItem`) | Default | Spell focus |
| `ItemThoughtKnot` | Default (`IotaHolderItem`) | Default | Thought knot |
| `ItemSpellbook` | Custom (writes to current page) | Default | Spellbook; writes to the currently open page |

### ItemSpellbook's Custom Writer

```kotlin
IOMethod(
    ItemSpellbook::class.java,
    writer = { stack, nbt ->
        val idx = ItemSpellbook.getPage(stack, 1)
        val pageKey = idx.toString()
        stack.getOrCreateCompound(ItemSpellbook.TAG_PAGES).put(pageKey, nbt)
    }
)
```

## Practical Examples

### Example 1: Register for an IotaHolderItem

If your item implements the `IotaHolderItem` interface, simply pass the `Class` object:

```kotlin
import io.yukkuric.hexparse.api.HexParseAPI

HexParseAPI.CreateItemIOMethod(
    cls = MyFocusItem::class.java
)
```

Both `writer` and `reader` use the default `IotaHolderItem` behavior.

### Example 2: Register for a Custom Storage Item

If your item uses a custom NBT structure to store Iotas:

```kotlin
import io.yukkuric.hexparse.api.HexParseAPI
import net.minecraft.nbt.CompoundTag
import net.minecraft.world.item.ItemStack

HexParseAPI.CreateItemIOMethod(
    cls = MyCustomStorageItem::class.java,
    writer = { stack, nbt ->
        stack.getOrCreateTag().put("stored_iota", nbt)
    },
    reader = { stack ->
        stack.tag?.getCompound("stored_iota")
    }
)
```

### Example 3: Use validator to Restrict Operations

```kotlin
import io.yukkuric.hexparse.api.HexParseAPI

HexParseAPI.CreateItemIOMethod(
    cls = MyConditionalItem::class.java,
    writer = { stack, nbt ->
        stack.getOrCreateTag().put("hex_data", nbt)
    },
    reader = { stack ->
        stack.tag?.getCompound("hex_data")
    },
    validator = { stack, isWrite ->
        if (isWrite) {
            // Only activated items can be written to
            stack.hasTag() && stack.tag!!.getBoolean("activated")
        } else {
            // Any state can be read from
            true
        }
    }
)
```

### Example 4: Use priority to Control Main/Offhand Preference

```kotlin
import io.yukkuric.hexparse.api.HexParseAPI

// High-priority item (lower priority number)
HexParseAPI.CreateItemIOMethod(
    cls = MyPriorityItem::class.java,
    priority = -100
)

// Normal-priority item
HexParseAPI.CreateItemIOMethod(
    cls = MyNormalItem::class.java,
    priority = 0
)
```

### Example 5: Register in Java

```java
import io.yukkuric.hexparse.api.HexParseAPI;
import net.minecraft.nbt.CompoundTag;
import net.minecraft.world.item.ItemStack;

HexParseAPI.CreateItemIOMethod(
    MyCustomItem.class,
    (stack, nbt) -> {
        stack.getOrCreateTag().put("iota_data", nbt);
    },
    (stack) -> {
        return stack.getTag() != null ? stack.getTag().getCompound("iota_data") : null;
    },
    0,
    null
);
```

## Important Notes

- Each `Class` can only have one `IOMethod` registered; duplicate registrations will output an error in the log (without overwriting)
- `IOMethod` automatically registers itself upon construction — no additional registration call is needed
- Do not modify the item's type or count in `writer` or `reader`; only modify NBT data
- `reader` returning `null` indicates the item currently has no stored Iota data
- Registration should be done during the mod initialization phase, ensuring it completes before players use the feature
