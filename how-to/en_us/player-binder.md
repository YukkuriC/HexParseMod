## Overview

`IPlayerBinder` is the player context injection mechanism in HexParse's parsing pipeline. When a parser needs access to the caster's information (e.g., entity references, permission checks, world state), implementing this interface allows it to automatically receive the current `ServerPlayer` reference before each parse.

## Interface Definition

```java
public interface IPlayerBinder {
    void BindPlayer(ServerPlayer p);
}
```

A single method ŌĆö `BindPlayer` ŌĆö automatically called by `ParserMain` before parsing begins.

## Invocation Timing

`ParserMain` binds the player at both parsing entry points:

- **Forward parsing** (`ParseCode`): Iterates all `str2nbtParsers` and calls `BindPlayer` on `IPlayerBinder` instances
- **Backward parsing** (`ParseIotaNbt`): Iterates all `nbt2strParsers` and calls `BindPlayer` on `IPlayerBinder` instances

This means whether your parser is `IStr2Nbt` or `INbt2Str`, as long as it also implements `IPlayerBinder`, the bound player reference will be available when `parse` executes.

## Usage

### Combined with IStr2Nbt

Have your forward parser implement `IPlayerBinder` alongside, cache the player reference in `BindPlayer`, and use it in `parse`:

```kotlin
class MyParser : BaseConstParser.Prefix("my_"), IPlayerBinder {
    private lateinit var player: ServerPlayer

    override fun BindPlayer(p: ServerPlayer) {
        player = p
    }

    override fun parse(node: String): CompoundTag {
        // Use player to access caster information
        val level = player.serverLevel()
        // ...
    }
}
```

### Combined with INbt2Str

The same applies to backward parsers:

```kotlin
class MyBackParser : INbt2Str, IPlayerBinder {
    private lateinit var player: ServerPlayer

    override fun BindPlayer(p: ServerPlayer) {
        player = p
    }

    override fun match(node: CompoundTag): Boolean { ... }

    override fun parse(node: CompoundTag): String {
        // Use player to access caster information
    }
}
```

## Important Notes

- `BindPlayer` is called **before every parse invocation** ŌĆö do not assume the reference remains unchanged across multiple parses
- Do not store long-term state in `BindPlayer` ŌĆö it is a "refresh" operation, not an "initialization" operation
- If your parser only needs player information under certain conditions, cache the reference in `BindPlayer` and use it conditionally in `parse`
- The `ServerPlayer` received by `BindPlayer` could theoretically be `null` (though not in normal casting flows); defensive checks are recommended
